# CLAUDE.md — OELTKit (Open E-Learning Toolkit)

> Copy this file to the root of the new code repo. It is the build instructions for agent sessions. Keep it accurate — stale conventions here cause compounding errors.

## Naming convention

Project/org/npm scope: **oeltkit** (`github.com/oeltkit`, `@oeltkit/*`). Short prefix **`oelt`** is used for the CLI binary (`oelt package`), custom element names (`<oelt-mcq>`), runtime global (`oelt.track`), and CSS tokens (`--oelt-*`). Never mix these up: packages are `@oeltkit/`, everything author-facing is `oelt`.

## What this project is

An open source, LLM-first e-learning toolkit: runtime + web components + packager + validators + MCP server, producing SCORM 1.2 / SCORM 2004 / cmi5 / standalone-web courses from a single course tree. Full context: `docs/PLAN.md`. Authoritative specs: `specs/`. **Specs win over code** — if implementation and spec disagree, the spec is right; fix the code or raise the conflict, never silently diverge.

## Repo layout

```
specs/          Behavior contracts. Versioned. Changes require human sign-off.
packages/
  runtime/      @oeltkit/runtime — tracking, state, navigation, a11y services
  components/   @oeltkit/components — custom elements
  cli/          @oeltkit/cli — package/preview/validate/new
  mcp/          @oeltkit/mcp — MCP server (wraps cli + fs operations)
harness/        Fake-LMS preview harness + SCORM/cmi5 test fixtures
examples/       Example courses (also serve as integration test fixtures)
docs/           Docs site source + llms.txt
```

## Hard rules

1. **No runtime dependencies** in `@oeltkit/runtime` and `@oeltkit/components` except `scorm-again` and `@xapi/*` (runtime only, bundled). Dev dependencies are fine. Adding any runtime dep requires human approval.
2. **Web platform only:** vanilla custom elements. No Lit, no React, no framework — anywhere in shipped code.
3. **TypeScript source → dual output:** plain ESM (`dist/esm/`) + single-file IIFE bundle (`dist/oelt.min.js`). Generated courses must work with no build step.
4. **Styling via design tokens only.** Components consume `--oelt-*` custom properties; never hardcode colors/sizes; never use Tailwind in toolkit code. Expose `::part()` for theme authors.
5. **Accessibility is a merge gate.** Every component: full keyboard operation, documented SR behavior (in the component's `README.md`), axe-core Playwright test passing, visible focus, `prefers-reduced-motion` respected. WCAG 2.2 AA.
6. **Suspend data is budgeted.** Components must declare their max state size; total enforced ≤ 3 KB (SCORM 1.2 headroom).
7. **Browser targets:** evergreen. No platform features newer than ~2 years past baseline availability.
8. **Never edit `imsmanifest.xml` / `cmi5.xml` templates ad hoc** — they are generated by `packages/cli` from `course.json`; fix the generator.

## Workflow for agent sessions

1. Read the relevant spec in `specs/` before touching a package.
2. Work on one package per session where possible; the only coupling point is `specs/manifest-v0.md`.
3. **Self-check before declaring done:** `npm test` (workspace), `npm run validate:examples` (runs `oelt validate` on every example course), and for component work `npm run test:a11y`.
4. Anything that changes observable behavior of a published surface → update the spec + docs + llms.txt in the same change, and flag it in the PR description as `SPEC CHANGE`.
5. Commits: conventional commits (`feat(components): …`). One logical change per PR. PRs include a "How I verified this" section.
6. If a spec is ambiguous, **stop and write the question into `specs/OPEN-QUESTIONS.md`** rather than guessing. Guessed semantics are the most expensive bug class in this project.

## Definition of done — component checklist

- [ ] Spec/contract in `specs/components/<name>.md` (behavior, attributes, events, a11y, state size)
- [ ] Implementation + unit tests
- [ ] Playwright test: keyboard-only operation path
- [ ] axe-core scan clean on demo page
- [ ] Demo page in `harness/demos/<name>.html`
- [ ] Tracking events verified in fake-LMS harness (statements visible & correct)
- [ ] `README.md` with 2–3 canonical usage examples (these get compiled into llms.txt — write them for model consumption)
- [ ] State round-trips through suspend/resume

## Testing infrastructure

- `harness/` serves any example course with a fake SCORM 1.2 / 2004 API and a cmi5 stub LRS; it renders a live panel of every tracking call. Use it (`npm run harness`) to verify behavior, including via screenshot.
- Real-LMS conformance: CI posts packages to SCORM Cloud (secrets in CI only). Don't attempt to replicate SCORM Cloud locally.
- An agent that hasn't run the harness on its change hasn't verified it.

## Things agents get wrong here — read twice

- SCORM 1.2 `lesson_status` conflates completion and success — the collapse rule lives in `specs/tracking-semantics.md`; never improvise it.
- `suspend_data` 4 KB limit (SCORM 1.2): always go through `runtime/state`, never call the SCORM API directly from components.
- ARIA: prefer native elements over ARIA roles; do not add `role`/`aria-*` without consulting the component spec — plausible-looking ARIA that breaks SR flows is the canonical LLM failure in this domain.
- cmi5 launch: `fetch` of the auth token happens once; statements require the exact context template from the launch parameters. Use `runtime/adapters/cmi5.ts` helpers only.
- Shadow DOM: content-bearing components use **light DOM** (author CSS must reach content); only chrome-like components use shadow DOM. Per-component decision recorded in its spec.
